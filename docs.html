Documentation

From zero to running conspiracy in about five minutes.

Prerequisites

  • An OCI-compatible container runtime — Apple Container (macOS), Podman, or Docker
  • Go 1.22+ (to build from source)
  • An OpenRouter API key — one key, access to every model

No Kubernetes. No cloud account. ConspiracyOS is a single container that runs anywhere you can run a systemd Linux container.

Install

ConspiracyOS is a Linux container — the container is the OS. It runs Ubuntu 24.04 with systemd as PID 1, and uses standard Linux primitives (users, ACLs, sudoers, nftables) to isolate and coordinate agents. conctl is the inner runtime baked into the container image. conos is the outer CLI you install on your host — it wraps SSH to conctl. Install both:

Build from source:

git clone https://github.com/ConspiracyOS/agent-runner.git
cd agent-runner
make image          # builds conctl binary + container image

# Also build the conos outer CLI
git clone https://github.com/ConspiracyOS/conos.git
cd conos && go build -o ~/bin/conos ./cmd/conos/

Set your API key and point conos at the instance:

echo "CONOS_API_KEY=sk-or-your-key-here" > srv/dev/container.env

# .conos/conos.toml — tells conos how to reach the instance
mkdir .conos && cat > .conos/conos.toml << 'EOF'
[instance]
host = "conos"   # SSH host alias — add to ~/.ssh/config
EOF

Bootstrap

Deploy the container. Bootstrap runs automatically on first boot as a systemd oneshot — no manual steps needed:

conos start

Bootstrap creates the full Linux environment inside the container:

  • Linux users a-concierge, a-sysadmin, etc. — one per agent
  • /srv/conos/agents/<name>/inbox/ — filesystem inbox for each agent
  • systemd path units that trigger agents when new tasks arrive
  • ACLs controlling which agents can task which

Bootstrap is idempotent — re-runs safely after config changes. Trigger it manually to apply a new config profile:

conos config apply

Your First Task

Drop a task into the outer inbox. The concierge routes it:

conos agent "What agents are available and what can they do?"

Watch it move through the system:

conos status              # agent status + pending inbox counts
conos agent logs -f       # follow the audit log

Runner: PicoClaw (default)

PicoClaw is a built-in Claude client compiled directly into the conctl binary. No installation needed. It talks to OpenRouter by default, giving you access to Claude, GPT-4, Gemini, and hundreds of other models with a single API key.

Configure it in /etc/conos/conos.toml:

[base]
runner = "picoclaw"
provider = "openrouter"
api_key_env = "CONOS_API_KEY"

# Different models per tier
[base.officer]
model = "anthropic/claude-opus-4-6"

[base.operator]
model = "anthropic/claude-sonnet-4.6"

The model value is the OpenRouter model ID. Browse available models at openrouter.ai/models.

You can override the provider or model per agent:

[[agents]]
name = "analyst"
tier = "worker"
provider = "openrouter"
model = "google/gemini-flash-1.5"   # cheap and fast for low-stakes tasks
api_key_env = "CONOS_API_KEY"

Runner: Exec (external CLI)

The exec runner pipes tasks into any external CLI binary via stdin and reads the response from stdout. This is how you plug in Claude Code, Codex, or any tool that accepts prompts from the command line.

The binary must be installed on the host — it is not baked into the image. Set runner to the binary name or path:

[[agents]]
name = "engineer"
tier = "worker"
runner = "claude-code"       # must be on PATH

Runner: Claude Code

Claude Code can act as an agent runner — it has tool use, file access, and long context. Install it inside the container and point an agent at it:

# Install into the container (run once)
container exec conspiracyos npm install -g @anthropic-ai/claude-code
[[agents]]
name = "engineer"
tier = "worker"
runner = "claude-code"
instructions = """
You are a software engineer. Implement tasks precisely.
Work inside your workspace directory.
"""

The agent's workspace at /srv/conos/agents/engineer/workspace/ becomes the working directory for the Claude Code process.

Runner: OpenAI Codex

# Install codex CLI
npm install -g @openai/codex

# Add API key
echo "OPENAI_API_KEY=sk-your-key" >> /etc/conos/env
[[agents]]
name = "coder"
tier = "worker"
runner = "codex"

Agent Tiers

Every agent belongs to a tier. Tier determines Linux group membership and what systemd hardening applies — officers get the most access, workers the least.

TierLinux groupTypical useHardening
officerofficersStrategy, policy, approval gatesBroad read/write, no NoNewPrivileges
operatoroperatorsRouting, coordination, system managementScoped read/write, NoNewPrivileges
workerworkersSpecific tasks — coding, analysis, writingStrictest — read-only system, workspace only

Agent Modes

on-demand — agent activates when a task appears in its inbox. A systemd path unit watches the inbox directory and starts the service on change.

cron — agent runs on a schedule. Use for morning briefings, daily reports, or periodic system checks.

[[agents]]
name = "reporter"
tier = "officer"
mode = "cron"
cron = "0 8 * * *"           # every day at 08:00
instructions = """
Review the audit log from the past 24 hours and summarize notable activity.
"""

Adding an Agent

Add a [[agents]] block to .conos/conos.toml, then apply the updated config to the running instance:

[[agents]]
name = "researcher"
tier = "worker"
mode = "on-demand"
instructions = """
You are a researcher. Given a topic, search the web and write a concise
summary with sources. Save your output to your workspace.
"""
conos config apply   # pushes config to instance + re-provisions
conos status         # verify it's live

Roles and Skills

Roles inject additional instructions and tool permissions into an agent. The sysadmin role, for example, gives the agent a library of skills for managing Linux users, services, and configs. Skills are plain Markdown files — readable by humans, injected into the agent's context at runtime.

[[agents]]
name = "sysadmin"
tier = "operator"
roles = ["sysadmin"]         # injects skills from configs/default/roles/sysadmin/
instructions = "..."

Skills are paired with sudoers entries — if a skill teaches the agent to run a command, the corresponding sudoers rule must allow it. The pair is the contract.

Configuration Reference

Config lives in .conos/conos.toml (project-local) or ~/.conos/conos.toml (global). Applied to the instance with conos config apply, which writes to /etc/conos/conos.toml inside the container. Resolution order: agent → base.<tier> → base.

[system]
name = "my-conspiracy"       # human label, used in logs

[base]
runner = "picoclaw"          # default runner for all agents
provider = "openrouter"
api_key_env = "CONOS_API_KEY"  # env var holding your API key

# Per-tier model overrides
[base.officer]
model = "anthropic/claude-opus-4-6"

[base.operator]
model = "anthropic/claude-sonnet-4.6"

[base.worker]
model = "anthropic/claude-haiku-4-5-20251001"

[dashboard]
enabled = true
port = 8080                  # status page (omit section to disable)

[contracts.system]
disk_min_free_pct = 10
mem_min_free_pct = 10
max_load_factor = 4.0
healthcheck_interval = "5m"

Per-agent fields (all optional, override base defaults):

FieldWhat it does
nameAgent name → Linux user a-<name>
tierofficer | operator | worker
modeon-demand | cron
cronCron expression (mode = cron only)
runnerOverride base runner for this agent
providerOverride LLM provider
modelOverride model for this agent
api_key_envOverride API key env var name
rolesRole directories to inject as skills
groupsAdditional Linux groups
max_sessionsMax concurrent sessions (default: 1)
instructionsSystem prompt for this agent

CLI Commands

conos is the outer CLI you run on your host. It wraps SSH to conctl inside the instance — you never invoke conctl directly.

CommandWhat it does
conos startBoot the instance.
conos stop [--force]Stop the instance (prompts for confirmation without --force).
conos statusShow each agent's systemd unit state and number of pending inbox tasks.
conos config applyPush config from .conos/conos.toml to the instance and re-provision.
conos agent <task>Send a task to the concierge (shorthand).
conos agent task <name> <task>Send a task directly to a named agent's inbox.
conos agent listList agents and their current state.
conos agent logs [-f] [-n N] [name]Show audit log — all agents or one. -f to follow.
conos agent kill <name>Stop a running agent.

Inspecting the System

Everything is on disk. No proprietary tooling needed.

# Agent status (systemd units + inbox counts)
conos status

# Follow the audit log
conos agent logs -f

# Tail a specific agent's log
conos agent logs -n 50 concierge

# List agents and state
conos agent list

For deeper inspection, SSH into the instance directly — the entire state tree is plain files:

ssh conos

# What's waiting in an inbox
ls /srv/conos/agents/concierge/inbox/

# Read a task file
cat /srv/conos/agents/sysadmin/inbox/001-abc.task

# Watch in real time
tail -f /srv/conos/logs/audit/$(date +%Y-%m-%d).log

# See active contracts
ls /srv/conos/contracts/

# Check agent filesystem layout
ls /srv/conos/agents/concierge/
# inbox/   outbox/   workspace/   sessions/

Contracts

The contracts binary is a standalone contract enforcement tool that works with any git project — no ConspiracyOS required. Define machine-readable invariants in YAML, run them on git hooks or a schedule, and optionally dispatch failures to any shell command.

Inside ConspiracyOS, the conctl healthcheck command uses the same contract engine to evaluate detective contracts on a systemd timer and escalate failures to sysadmin.

Install the standalone binary:

go install github.com/ConspiracyOS/contracts/cmd/contracts@latest

cd your-project
contracts init    # creates .agent/contracts/ + installs git hooks
contracts audit   # run all contracts

Contract Schema

Contracts live in .agent/contracts/*.yaml. Each file defines one contract with one or more checks:

id: C-SEC-001
description: No private key material in committed files
type: atomic
trigger: commit
scope: global
checks:
  - name: no private key headers
    no_regex_in_file:
      pattern: "-----BEGIN (RSA |EC )?PRIVATE KEY-----"
    on_fail: escalate

Check modules: command, script, regex_in_file, no_regex_in_file, path_exists, path_not_exists, yaml_key, json_key, toml_key, env_var, no_env_var, command_available.

on_fail values: fail (default), warn, require_exemption, escalate, alert, halt_agents.

Built-in contracts for common stacks (Go, TypeScript) are activated by declaring stack in .agent/config.yaml.

See the contract repo for the full schema reference.

Contract Escalation

When a check with on_fail: escalate or on_fail: alert fails, the binary shells out to a user-configured command with a JSON payload on stdin. Configure it in .agent/config.yaml:

escalation:
  command: "curl -sf -X POST $WEBHOOK_URL -H 'Content-Type: application/json' -d @-"

The command receives a JSON payload on stdin:

{
  "severity": "escalate",
  "summary": "1 escalate failure(s): C-SEC-001",
  "failures": [{ "contract_id": "C-SEC-001", "check_name": "no private key headers",
                  "status": "fail", "on_fail": "escalate", ... }]
}

Env vars CONTRACTS_SEVERITY and CONTRACTS_SUMMARY are also set. The command exit code does not affect the audit exit code — escalation is best-effort. escalate severity beats alert when both are present.
