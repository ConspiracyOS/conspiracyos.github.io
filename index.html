A conspiracy of agents.

They run on Linux, not faith.

Most agent frameworks control AI with application logic — permission configs, trust levels, guard clauses in code. ConspiracyOS uses Linux. Each agent is a Linux user. Its permissions are chmod. Its network is nftables. Its process boundaries are systemd. A fully compromised agent still can't exceed what the OS allows.

That's not a design goal. That's stat(2).

Philosophy

Enforced, not promised. Other systems tell the model not to do things. ConspiracyOS uses sudoers, nftables, and chmod. If the capability doesn't exist in Linux, it doesn't exist. A permission config that the agent itself can edit isn't a permission.

Fine-grained, not binary. Most agent sandboxes are on/off. ConspiracyOS gives you the same access control model you use for system processes: per-uid file permissions, filesystem ACLs, sudoers allowlists, per-uid nftables rules. Officers, operators, and workers each get exactly what they need — enforced by the kernel, not by instructions.

Everything on disk. Inboxes are directories. Tasks are files. State survives reboots. Debug with ls, tail, stat — no proprietary tooling, no black boxes.

Self-healing. Detective contracts run on systemd timers. If the system drifts from its intended state, the sysadmin agent is notified. The conspiracy watches itself.

Runs anywhere. Any Linux machine with systemd. No cloud account. No container orchestrator. No vendor lock-in.

How It Works

A conspiracy is a fleet of agents coordinated using Linux OS primitives. Every agent is a Linux user. They communicate by writing files to each other's inboxes. All state is on disk.

Agents, Inboxes, Tiers

Agents are organized into three tiers. Tier determines Linux group membership and systemd hardening — officers get the most access, workers the least.

TierLinux groupWhoAuthority
officerofficersceo, cto, csoSets policy, approves elevated actions
operatoroperatorsconcierge, sysadminRoutes tasks, operates the system
workerworkers(task-specific)Bounded execution, spawned on demand

To send a task to another agent, write a plain text file to their inbox: /srv/conos/agents/<name>/inbox/<NNN>-<id>.task

Linux as Coordination Layer

  • Filesystem — inboxes, workspaces, artifacts, audit logs
  • systemd — path watchers trigger agents when inboxes change; timers run scheduled agents
  • ACLs — enforce who can write to whose inbox
  • nftables — per-uid rules enforce what each agent can reach on the network
  • sudoers — allowlist what each agent can execute as root

Inspectability

Every significant action is logged to /srv/conos/logs/audit/. Every capability is enforced by the OS, not by instructions. You can see exactly what's happening with standard tools:

# See what agents exist
ls /srv/conos/agents/

# See what's in an agent's inbox
ls /srv/conos/agents/concierge/inbox/

# Follow the audit log
tail -f /srv/conos/logs/audit/$(date +%Y-%m-%d).log

# See active contracts
ls /srv/conos/contracts/

Start a Conspiracy

Build the image, write your API key, start:

git clone https://github.com/ConspiracyOS/agent-runner.git
cd agent-runner
make image   # builds conctl binary + container image

echo "CONOS_API_KEY=sk-or-your-key-here" > srv/dev/container.env
conos start  # boots instance; bootstrap runs automatically on first start

Bootstrap creates the Linux users, directories, and systemd units. Drop a task and watch it move through the system:

conos agent "What agents are available and what can they do?"
conos status
